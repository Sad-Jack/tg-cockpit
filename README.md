# tg-cockpit

Переиспользуемый Telegram-кокпит для каналов **и** групп: вся работа с
Telegram через CLI (на user-сессии Telethon) плюс Telegram-бот-мост к headless
Claude Code. Управляешь любым числом каналов/групп из одной точки — историей,
аналитикой, постами, расписанием, опросами, правками, ответами в группах и
медиа. AI (Claude Code) — это «мозг» сверху, а не крутящийся 24/7 сервис.

_Стек: Python 3.12+ · Telethon (user-сессия) · aiogram (бот-мост) · Claude Agent SDK · Typer (CLI) · pytest. Лицензия — MIT._

## Что это и архитектура

Три слоя, каждый работает сам по себе:

1. **`tgcockpit` (CLI, channel-agnostic)** — весь обмен с Telegram через
   user-сессию Telethon: история и метрики, посты и серверное расписание,
   опросы и квизы, редактирование/удаление сообщений, чтение групп и ответы в
   них, медиа (до 10 картинок в посте), экспорт. Никакого AI — это просто
   надёжные «руки». Запускается как `uv run tgcockpit <команда>`.
2. **`channels/<name>/`** — отдельная папка на каждую сущность (канал или
   группа): свой `CLAUDE.md` (накопленная память — что знаем, выученные уроки,
   snapshot метрик), `config.yaml`, профиль голоса `skills/voice.md`, черновики,
   инсайты, контент-план и vault постов. Сущности изолированы — один
   воркспейс = один канал/группа.
3. **Claude Code + Skills (мозг)** — Telegram-бот принимает запрос (текст или
   голос) и передаёт его в headless-инстанс Claude Code через
   `claude-agent-sdk`. Claude читает память сущности, дёргает CLI как
   инструменты и обновляет память. Бот сам по себе «не умный» — он только мост.

**Инструкции агента — слоями, без дублей.** Каждое правило живёт в одном месте:
[`CLAUDE.md`](CLAUDE.md) — контракт (глобальные правила + рабочий цикл);
[`COMMANDS.md`](COMMANDS.md) — справочник CLI; `channels/NAME/CLAUDE.md` + `voice.md` —
факты конкретной сущности; `.claude/skills/*` — пошаговые процедуры, которые
ссылаются на контракт, а не пересказывают его.

**Линчпин:** серверные отложенные сообщения Telegram публикуются, даже если Mac
выключен (лимит **100 на чат, до года вперёд**). Поэтому автопостинг работает
локально, без сервера: запланировал с ноута → закрыл крышку → Telegram постит
сам. Дойдёшь до 101-го запланированного сообщения — лимит достигнут: чтобы
добавить новое, сначала отмени одно из уже стоящих в очереди.

## Phase 0 — получить доступы (руками, ~10 минут)

Без этого ничего сетевого не заработает.

1. **`api_id` + `api_hash` (обязательно):** зайти на <https://my.telegram.org> →
   *API development tools* → создать приложение. Скопировать `api_id` и
   `api_hash`. Это нужно для user-сессии (история, постинг, расписание, статы).
2. **`bot_token` (для бота-моста):** написать
   [@BotFather](https://t.me/BotFather) → `/newbot` → получить токен. Нужен,
   только если хочешь управлять кокпитом через Telegram-бота.
3. **Заполнить секреты:**
   ```bash
   cp .env.example secrets/.env
   # отредактировать secrets/.env: вписать API_ID / API_HASH (+ BOT_TOKEN для бота)
   ```
   Телефон и пароль 2FA вводятся один раз интерактивно при `auth` — в файлы не
   пишутся.

> `secrets/` целиком в `.gitignore`. Файл сессии = полный доступ к аккаунту,
> относись к нему как к приватному SSH-ключу.

## Установка

```bash
uv sync --extra bot          # .venv + зависимости + бот-мост (aiogram + claude-agent-sdk)
```

Дополнительные extras по необходимости:

```bash
uv sync --extra dev          # + pytest для тестов
uv sync --extra audio        # + faster-whisper (локальный фолбэк транскрибации)
uv sync                      # только базовый CLI, без бота
```

### Тесты

```bash
uv run --extra dev --extra bot python -m pytest -q
```

## Запуск

**Через бота (повседневный сценарий):**

- Двойной клик по `start-bot.command` в Finder — откроет Терминал, при
  необходимости проведёт разовый вход в Telegram и поднимет бота-мост.
- Или из терминала: `uv run tgcockpit bot`.

**Напрямую через CLI (без бота):**

```bash
uv run tgcockpit auth                          # одноразовый логин (телефон + код + 2FA)
uv run tgcockpit whoami                         # проверка: печатает текущий аккаунт
uv run tgcockpit channel init --channel mychan --handle @mychan
uv run tgcockpit study --channel mychan --full  # фетч + аналитика + vault + снимает гард
uv run tgcockpit drafts new --channel mychan --title "Привет"
uv run tgcockpit scheduled list --channel mychan
uv run tgcockpit schedule --channel mychan --draft channels/mychan/drafts/<file>.md --at "2026-06-10T09:00"
```

Полный гид по командам и флагам — в [`COMMANDS.md`](COMMANDS.md).

> **Что значит «изучена».** Сущность считается изученной, когда `study` отработал:
> история сфетчена, посчитана аналитика, собран vault и выведен профиль голоса —
> в `config.yaml` встаёт `studied: true` и снимается гард на постинг. `study`
> стоит периодически перезапускать: он подхватывает новые посты, вышедшие после
> прошлого прогона, и освежает память и профиль голоса.

## Как пользоваться через бота

Открываешь чат с ботом и работаешь голосом или текстом:

1. **`/menu`** — бот показывает список сущностей. Выбираешь канал/группу или
   добавляешь новую (бот авто-обнаруживает каналы/группы, где он сам админ;
   нужного нет — добавь бота админом со ВСЕМИ правами и снова открой `/menu`).
2. **Пишешь задачу обычным языком** — текстом или голосовым. Голос
   расшифровывается автоматически (см. ниже). Например: «сделай аналитику за
   месяц», «набросай пост про релиз с картинкой», «поставь это на четверг 9
   утра», «создай опрос про время эфира», «ответь в группе на сообщение про
   доставку».
3. **Бот стримит этапы** — какие инструменты дёргает Claude, что думает, что
   получилось. А в терминале, где запущен бот, виден полный «живой» поток агента
   (запрос → размышления → вызовы инструментов → ответ), как в обычном CLI Claude Code.
4. **Любая публикация ждёт твоего подтверждения кнопкой.** Публикующие команды
   (`post`, `schedule`, `message edit/delete`, `poll create`, `group reply`) не
   выполняются сразу — Claude готовит действие, а ты жмёшь кнопку в Telegram, сам
   агент не публикует (полные правила для агента — CLAUDE.md).

> **Гард изучения:** создание контента (`post`, `schedule`, `poll create`,
> `group reply`) заблокировано, пока сущность не изучена (`study`); зайдёшь в
> неизученную — бот предложит изучить (полные правила для агента — CLAUDE.md).

## Безопасность

### Доступ к боту — настрой ОБА уровня

Бот управляет твоими каналами и группами, поэтому никто, кроме тебя, не должен с
ним взаимодействовать. Защита двойная:

1. **Нативно в Telegram (рекомендуется!):** [@BotFather](https://t.me/BotFather)
   → твой бот → **Bot Settings → Allowed Users → Restrict bot usage** (вкл) →
   **Allowed users: You**. После этого Telegram сам не пускает к боту никого,
   кроме тебя — сообщения чужих даже не доходят. Самый надёжный барьер.
2. **В коде (по умолчанию, без настройки):** бот отвечает только владельцу — его
   Telegram id берётся **автоматически из твоей user-сессии** (`API_ID/API_HASH`).
   Чужим — «⛔ Доступ запрещён». Доп. пользователей можно добавить через
   `TGCOCKPIT_OWNER_IDS` (через запятую) — обычно не нужно.

### Секреты и публикация

- Секреты (`api_id`, `api_hash`, `.session`, токен бота, ключ Anthropic) живут
  **только** в `secrets/` (в `.gitignore`). В папках каналов секретов нет.
- `.session` — SQLite с полным доступом к аккаунту → держи `chmod 600`, никогда
  в git. Утечка сессии = угон аккаунта. Для интенсивной автоматизации рассмотри
  отдельный аккаунт.
- **Публикация — только по кнопке.** Любое публикующее действие через бота
  требует подтверждения кнопкой в Telegram. Агент не публикует сам.

## Транскрибация голоса

Голосовые в бота расшифровываются цепочкой бэкендов (первый успешный побеждает):

1. **Telegram (Premium)** — серверный `messages.transcribeAudio` через
   user-сессию. Точно, без локальных моделей; Premium — без лимитов.
2. **Apple** — on-device через Shortcuts (`TgCockpitTranscribe`) или
   SFSpeechRecognizer. Оффлайн.
3. **Whisper** — локальный фолбэк `faster-whisper` (extra `audio`, CPU int8,
   читает `.ogg` напрямую).

CLI-команда для разовой расшифровки файла:
`uv run tgcockpit transcribe --file ./voice.m4a --lang ru`.

## Obsidian-хранилище постов

`study` (и команда `vault`) собирают vault постов сущности в
`channels/<name>/<@handle>/`: каждый текстовый пост = отдельный `.md` + общий
`index.md`. Obsidian не обязателен — это просто папка `.md`-файлов; открыть её в
Obsidian опционально (за удобную навигацию по заметкам), но всё работает и без
него. Офлайн, только текст: медиа, опросы и аудио в vault не попадают, потому что
vault хранит исключительно текстовое тело постов (картинки/файлы/опросы — не
текст, сохранять их как `.md` нечего). Пересобрать из кэша истории без сети —
`uv run tgcockpit vault --channel NAME`.

## Оговорки

- Автоматизация user-аккаунта — серая зона ToS Telegram. Держи объём
  «человеческим»: рекомендуем ≤10 постов в день и ≤100 запланированных вперёд.
- Всё работает **локально, без всегда-включённого сервера**. Серверное
  планирование стреляет только заранее заготовленными постами в фиксированное
  время; реакция на живые события при выключенном Mac невозможна без always-on
  хоста (вне скоупа).
- Broadcast stats (Tier-2) доступны только крупным каналам — Tier-1 (из кэша,
  без сети) самодостаточен.
