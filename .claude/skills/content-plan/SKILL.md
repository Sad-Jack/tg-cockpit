# /content-plan — план следующих N постов (БЕЗ СЕТИ)

Цель: на основе «мозга» сущности и последних инсайтов предложить N тем и записать их в `channels/NAME/content-plan.md`. Никаких обращений к Telegram (ни `scheduled list`, ни `fetch-history`, ни `analytics`). Только чтение локальных файлов и запись плана.

Это лёгкая задача и она ЗДЕСЬ заканчивается: показал темы списком — остановился. НЕ переходи дальше к `/draft-post`, `drafts new` или `schedule`, пока пользователь явно не попросит написать/поставить (см. «Не выходи за рамки просьбы» в CLAUDE.md). Темы должны ДОПОЛНЯТЬ то, что уже есть в канале (читай `content-plan.md` и память) и ложиться в столпы — не плодить дубли уже вышедшего/запланированного. Если тема держится на личной истории автора (его путь, опыт, рефлексия) — пометь «нужна твоя история»: такую сам не пиши, дождись фактов от пользователя.

Гард изучения (см. CLAUDE.md): post/schedule/poll create/group reply требуют studied: true; если false — предложи /study-entity и остановись. Даже план опирается на изученные данные — без `study` строить нечего.

## Вход
- `NAME` — имя сущности (воркспейс канала ИЛИ группы, папка `channels/NAME/`).
- `N` — сколько постов предложить. Если N не задано — частота постинга × 1–2 недели или по слотам `config.yaml`.

## Шаги

1) Прочитать «мозг» сущности: `channels/NAME/CLAUDE.md`.
   Извлечь:
   - **Столпы (pillars)** — для распределения тем.
   - **Частота и расписание постинга** — слоты (например, `mon 09:00`), частота.
   - **Лучшее время** — когда реакция выше.
   - **Гипотезы** из аналитики — что предстоит проверить (формат, угол, время, длина, тема).
   Заодно прочитать `channels/NAME/skills/voice.md` (стиль/тон), чтобы формулировки тем были в голосе сущности. При необходимости свериться с `channels/NAME/config.yaml` (`pillars`, `timezone`, слоты).

2) Прочитать последний файл инсайтов, если есть: взять самый свежий по дате из `channels/NAME/insights/<date>.md`.
   Если файлов нет — пропустить шаг, опираться только на CLAUDE.md.
   Использовать инсайты для приоритизации тем (что заходило, что нет, какие столпы недогружены, какие гипотезы свежие).

3) Предложить N тем. Для каждой:
   - привязать к **столпу** (из списка столпов): распределяй по дефициту — какой столп меньше всего запланирован в ближайшие N недель (по `content-plan.md` + инсайтам), тот идёт первым; затем корректируй инсайтами (что заходило, какие гипотезы свежие);
   - привязать к конкретному **слоту/дате** из расписания, начиная с ближайшей будущей даты от текущей (системная дата), учитывая лучшее время и timezone;
   - привязать к **ГИПОТЕЗЕ** из аналитики/инсайтов — указать, что именно проверяем (план должен двигать гипотезы, а не быть случайным);
   - формулировать темы коротко и в голосе сущности.

4) Записать результат в `channels/NAME/content-plan.md` — markdown-таблица:

   content-plan.md — внутренний файл памяти, таблица здесь допустима и в Telegram НЕ публикуется (исключение из правила таблиц — см. CLAUDE.md).

   | Дата | Слот | Столп | Тема | Статус |
   |------|------|-------|------|--------|
   | 2026-06-09 | mon 09:00 | <pillar> | <тема + гипотеза, что проверяем> | planned |

   - `Статус` новых строк — `planned`.
   - Не затирать уже запланированные/опубликованные строки — дополнять/обновлять существующую таблицу. Сортировать по дате/слоту.

## Запрещено
- Любые сетевые команды CLI (`fetch-history`, `analytics`, `post`, `schedule`, `scheduled list/cancel`, `export`) — Telegram НЕ дёргается.
- Создание черновиков (`drafts new`) — этот skill только планирует темы в `content-plan.md` (тело постов пишет `draft-post`).
- Выдумывать столпы/слоты: брать строго из `CLAUDE.md` / `config.yaml`.

## Выход
Обновлённый `channels/NAME/content-plan.md` с таблицей и краткая сводка фиксированной структуры:
- N тем;
- по каким столпам;
- даты X–Y (от ближайшей до последней);
- какие гипотезы двигаем;
- сколько строк новых / обновлённых.

Дисциплина памяти — см. /update-memory.
