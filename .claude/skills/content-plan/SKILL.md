# /content-plan — план обучающих постов (БЕЗ СЕТИ)

Цель: предложить очередь **обучающих постов** (эксплейнеры, разборы — «X на пальцах», «что такое X», «как работает X») и записать их в `channels/NAME/content-plan.md`. Заранее планируется только это.

**Что НЕ входит в content-plan:** личные истории / дневник, анонсы, продуктовые и релизные посты. Их пользователь приносит сам по мере появления («хочу пост про X, вот что расскажу») — не генерируй их в долгосрок и не вставляй в план. Никаких «столпов» и ротации по ним: план — это один поток обучающих тем.

Никаких обращений к Telegram (ни `scheduled list`, ни `fetch-history`, ни `analytics`). Только чтение локальных файлов и запись плана.

Это лёгкая задача и она ЗДЕСЬ заканчивается: показал темы — остановился. НЕ переходи к `/draft-post`, `drafts new` или `schedule`, пока пользователь явно не попросит написать/поставить (см. «Не выходи за рамки просьбы» в CLAUDE.md).

Гард изучения (см. CLAUDE.md): требует studied: true; если false — предложи /study-entity и остановись.

## Вход
- `NAME` — имя сущности (папка `channels/NAME/`).
- Горизонт / `N` — на какой срок или сколько постов. «На месяц» → разложи по слотам `config.yaml` до конца текущего месяца. Если не задано — частота постинга × 1–2 недели.

## Шаги

1) Прочитать память: `channels/NAME/CLAUDE.md` (что за канал, какие обучающие темы уже выходили — чтобы не плодить дубли), `channels/NAME/skills/voice.md` (тон формулировок), `channels/NAME/config.yaml` (слоты, timezone). Глянуть уже существующий `content-plan.md` (что стоит/в бэклоге) и самый свежий `insights/<date>.md`, если есть.

2) Подобрать обучающие темы:
   - только те, что ещё НЕ выходили и не стоят в плане (без дублей);
   - выстраивай в логичную последовательность: от простого к сложному, зависимые темы — после базовых (напр. эмбеддинги → затем RAG);
   - формулируй коротко и в голосе канала;
   - свежие гипотезы из инсайтов учитывай (формат/длина/время), но это поток обучающих постов, а не личное/анонсы.

3) Разложить по датам: по слотам из `config.yaml`, начиная с ближайшей будущей даты (системная дата), по timezone канала, до конца горизонта.

4) Записать в `channels/NAME/content-plan.md` — чистый планер (внутренний файл памяти, таблица допустима, в Telegram НЕ публикуется):
   - короткая шапка: ритм/слот + легенда статусов;
   - таблица месяца, отсортированная по дате: `| Дата | День | Тема | Статус |` (только обучающие посты, без колонки «столп»);
   - раздел «Бэклог идей» — обучающие темы без даты.

   Статусы: `идея → план → черновик → в очереди → вышел`. Новые датированные → `план` (или `черновик`, если черновик уже есть). Не затирай существующие строки (особенно `в очереди`/`вышел`) — дополняй и пересортировывай по дате.
   НЕ засоряй файл: без эссе, «смысловых линий», стале-пометок «ждёт ✅», без личных/анонсовых тем. План читается как план.

## Запрещено
- Сетевые команды CLI (`fetch-history`, `analytics`, `post`, `schedule`, `scheduled list/cancel`, `export`).
- Создание черновиков (`drafts new`) — скилл только планирует темы (тело пишет `draft-post`).
- Вставлять в план личные истории, дневник, анонсы, релизы — это ad-hoc контент пользователя, не для долгосрочного плана.

## Выход
Обновлённый `channels/NAME/content-plan.md` (месячная таблица обучающих постов + бэклог) и краткая сводка: сколько тем, на какие даты (X–Y), сколько строк новых / обновлённых.

Дисциплина памяти — см. /update-memory.
