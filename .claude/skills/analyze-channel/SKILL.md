# /analyze-channel

Цель: обновить аналитику сущности (канала ИЛИ группы), записать в память только **дистиллят** (выводы + ключевые числа) и **сформулировать проверяемые гипотезы** для следующего цикла. Раскладка хранилища: `insights/` держит `<date>.md` (читаемый отчёт) + `<date>.json`; `data/` держит `history.json` + сырьё. Ни то, ни другое не копируется в `CLAUDE.md` — туда идёт только короткий Snapshot. Это экономит токены: в следующих сессиях агент читает память, а не сырой JSON.

Подставь имя сущности вместо `NAME`.

## Чек-лист действий

1. **Определить тип сущности.** Прочитай `channels/NAME/config.yaml`, поле `kind`:
   - `channel` → анализ контента (время, столпы, медиа, длина).
   - `group` / `supergroup` → анализ сообщества (активность, вовлечённость, темы).
   - Дальнейшие акценты (шаг 5) зависят от `kind`.

2. **Собрать историю** (инкрементально — дешевле по токенам и времени):
   ```
   uv run tgcockpit fetch-history --channel NAME
   ```
   - Первый сбор для сущности **или** осознанный полный рефреш — добавь `--full`:
     ```
     uv run tgcockpit fetch-history --channel NAME --full
     ```
   - Результат → `channels/NAME/data/history.json` (сырьё, в память НЕ идёт). `--full` — дорогая операция, без нужды не вызывай.

3. **Посчитать аналитику:**
   ```
   uv run tgcockpit analytics --channel NAME
   ```
   - Создаёт `channels/NAME/insights/<date>.md` (читаемый отчёт) и `channels/NAME/data/*.json` (сырые метрики).

4. **Прочитать свежий отчёт.** Возьми самый свежий по дате файл из `insights/`, напр. `ls -t channels/NAME/insights/*.md | head -1`, открой его и вытащи факты для следующих шагов.

5. **Оценить по типу сущности.** Количественные определения: «лучший слот» = топ-3 по медианному охвату; «проседает» = нижние 25% по ER; «повторяющаяся тема» = встречается в 3+ тредах.
   - **Канал** (`kind: channel`):
     - Лучшее время/слоты — когда охват и ER выше (топ-3 по медианному охвату).
     - Контент-столпы — какие `pillars` заходят, какие проседают (нижние 25% по ER).
     - Медиа — влияние картинок/без картинок на охват и реакции.
     - Длина — корреляция длины поста с вовлечённостью.
   - **Группа** (`kind: group|supergroup`):
     - Активность — пики/спады, кто и когда пишет.
     - Вовлечённость — доля отвечающих, длина тредов, реакции.
     - Темы — повторяющиеся (в 3+ тредах) вопросы и боли участников (пригодится для контент-плана канала).

6. **Сформулировать 2–4 проверяемые ГИПОТЕЗЫ.** Каждая — в формате причинно-следственного предсказания с метрикой и горизонтом проверки:
   - Шаблон: «Если делать **X** — вырастет **Y** на **Z**, проверим за **N** постов (или за период)».
   - BAD: «изображения помогают» — нет метрики, порога и горизонта, проверить нельзя.
   - GOOD: «+1 изображение → ER +15%, проверим за 6 постов» — есть направление, порог и горизонт.
   - Ещё примеры GOOD:
     - «Если постить столп *кейсы* в слот 09:00 — средний охват вырастет на ~15%, проверим за 5 постов.»
     - «Если сокращать посты до 600 знаков — дочитываемость/реакции вырастут на ~10%, проверим за 4 поста.»
   - Каждая гипотеза должна быть фальсифицируема: метрика + порог/направление + горизонт. Непроверяемые сейчас (нет данных/метрики) — отложи, не записывай как гипотезу.

7. **Дистиллировать в память** `channels/NAME/CLAUDE.md` (редактируй ТОЛЬКО нужные секции, не переписывай файл целиком):
   - **Snapshot** — обнови короткие ключевые числа под тип сущности:
     - канал: подписчики, средний охват/пост, ER, лучший слот, тренд за период;
     - группа: участники, активных/период, средняя длина треда, доля отвечающих, топ-темы.
     - 1–6 строк, без таблиц сырого JSON.
   - **Выученные уроки** — добавь 1–3 **датированные** строки (`YYYY-MM-DD: вывод`) с конкретными наблюдениями.
   - **Гипотезы** — запиши сформулированные на шаге 6 гипотезы (датируй, чтобы потом сверять). Их позже замкнёт `review-performance`: факт vs предсказание.

## Правила

- Дисциплина памяти — см. /update-memory.
- `data/*.json` и `insights/<date>.md` — источник деталей; ссылайся на них, не переноси.
- Гипотеза без метрики и горизонта проверки — не гипотеза; не записывай расплывчатые формулировки.
- `--full` — применяй при первом сборе или осознанном рефреше; иначе обычный инкрементальный `fetch-history`.
